# Agent Instructions: HTTP Deprecation

This file provides instructions for AI agents working with this Zuplo example.

## Before You Start

This example can run **without** a Zuplo account or portal setup. The deprecation feature uses a built-in outbound policy and does not require API keys or environment variables. Linking (`npx zuplo link`) is optional; the gateway and deprecation headers work the same with or without it.

If the user wants to **deploy** to Zuplo or use other Zuplo services, they will need a Zuplo project and may need to complete portal steps (e.g. environment variables for production).

## Setup Workflow

### Phase 1: Agent-Executable (Automated)

Run these commands to prepare and run the example locally:

```bash
# 1. Install dependencies
npm install

# 2. Start dev server (no .env or zuplo link required for deprecation)
npm run dev
```

The server will start on `http://localhost:9000`. All `/todos` routes will return responses with `Deprecation`, `Sunset`, and `Link` headers.

### Phase 2: Human-Required (Only If User Wants to Link or Deploy)

**If the user wants to link to Zuplo or deploy**, stop and inform them:

- **To link locally**: They must run `npx zuplo link` and complete the interactive prompts (account, project, environment). This creates `.env.zuplo`.
- **To deploy**: They need a Zuplo project. After `zuplo deploy`, any required environment variables must be set in the Zuplo Portal under Settings → Environment Variables.

Agents cannot run `zuplo link` interactively or create portal resources.

### Phase 3: Agent-Executable (Testing)

Once the dev server is running, agents can test without API keys:

```bash
# Full response with headers
curl -i http://localhost:9000/todos

# Headers only
curl -s -D - -o /dev/null http://localhost:9000/todos

# POST
curl -i -X POST http://localhost:9000/todos \
  -H "Content-Type: application/json" \
  -d '{"title": "Test", "completed": false, "userId": 1}'
```

Verify response headers include `deprecation: true`, `sunset`, and `link` with `rel="deprecation"`.

## File Modification Guidelines

### Safe to Modify

| File | Purpose | Notes |
|------|---------|-------|
| `config/routes.oas.json` | API routes | Add/modify endpoints; add or remove `http-deprecation-outbound` from `policies.outbound` |
| `config/policies.json` | Policy definitions | Change `http-deprecation-outbound` options (`deprecation`, `link`, `sunset`); add more outbound policies |
| `modules/*.ts` | Custom handlers/policies | Example uses built-in policy; custom modules are optional |

### Do Not Modify

| File | Reason |
|------|--------|
| `.env.zuplo` | Generated by `zuplo link`, contains auth tokens |
| `zuplo.jsonc` | Project metadata, rarely needs changes |

## Code Patterns

### Adding deprecation to a new route

In `config/routes.oas.json`, ensure the route has an outbound policy array that includes the policy name:

```json
"x-zuplo-route": {
  "handler": { ... },
  "policies": {
    "outbound": ["http-deprecation-outbound"]
  }
}
```

### Changing sunset date or migration link

In `config/policies.json`, find the policy with `"name": "http-deprecation-outbound"` and edit `handler.options`:

```json
"options": {
  "deprecation": true,
  "link": "https://example.com/your-migration-docs",
  "sunset": "2026-06-30T23:59:59Z"
}
```

Use ISO 8601 for `sunset`; Zuplo will send it as HTTP-date in the response.

### Adding a second deprecation policy (e.g. different per route)

In `config/policies.json`, add another policy with a different name and options:

```json
{
  "handler": {
    "export": "HttpDeprecationOutboundPolicy",
    "module": "$import(@zuplo/runtime)",
    "options": {
      "deprecation": true,
      "link": "https://example.com/v1-sunset",
      "sunset": "2025-12-31T23:59:59Z"
    }
  },
  "name": "http-deprecation-v1-outbound",
  "policyType": "http-deprecation-outbound"
}
```

Then reference `"http-deprecation-v1-outbound"` in the desired route’s `policies.outbound` array.

## Deployment

After changes are tested locally:

```bash
zuplo deploy
```

If the deployed project needs environment variables, the user must set them in the Zuplo Portal under Settings → Environment Variables. This example does not require env vars for the deprecation feature itself.

## Common Issues

| Symptom | Likely Cause | Agent Action |
|---------|--------------|--------------|
| No deprecation headers in response | Policy not on route or wrong policy name | Check `routes.oas.json`: `policies.outbound` must include the policy name that exists in `policies.json` |
| Wrong sunset format in config | Non-ISO date | Use `"2025-06-30T23:59:59Z"` in options |
| Backend 4xx/5xx | Todo backend unreachable | Deprecation headers are still added; suggest user check network or backend URL in route |
| ECONNREFUSED | Dev server not running | Run `npm run dev` |
